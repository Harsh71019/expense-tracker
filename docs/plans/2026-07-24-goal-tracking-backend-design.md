# Goal tracking ‚Äî backend (savings goals ‚Äî e.g. "Emergency Fund", "New Laptop")

## Context

TreasuryOps has zero goal-tracking code today ‚Äî `docs/EXTENSION-MODULES.md` ¬ß2 sketches a "Goals" design, but it's aspirational and scopes goal *feasibility* against salary projection + committed outflows (EMIs, budgets, SIPs) ‚Äî none of which exist in the codebase (`apps/api/src` has no `salary/`, `budgets/`, or `loans/` module; debt today lives only as `loan_liability`/`loan_receivable` asset kinds with manual valuations). Building the full feasibility calculator would mean building those prerequisite modules first.

Agreed scope (from prior discussion):
- **Standalone tracker now**: target amount + optional target date + live progress + a simple "at current contribution rate" projection, using only data that already exists (accounts, transactions). No dependency on salary/budgets/loans. Full feasibility-vs-cashflow math is an explicit later layer, not this pass.
- **Both funding modes**: `linked_account` (progress = a dedicated account's balance) and `tagged` (progress = sum of transactions carrying a goal tag) ‚Äî both per the original design doc.
- **Goal-achieved notifications** are in scope (new `notification_type`, nightly cron), reusing the outbox pattern that's already `@Global()`-wired specifically anticipating this (`apps/api/src/notifications/notifications.module.ts:11-13`: *"budgets/recurring/goals (Phase 4) will call NotificationOutboxRepository.enqueue(...)"*).

This document is a design only; implementation has not started. It is the backend half of a two-PR pair ‚Äî a sibling PR covers the frontend and targets the API shapes defined here.

## Key design decision: reuse existing primitives, add exactly one new table

Per `EXTENSION-MODULES.md`'s own stated stance ‚Äî *"model it as ledger mechanics, never a parallel money system... goals are views over the ledger plus small metadata"* ‚Äî and confirmed by exploration:
- **`linked_account` mode needs zero transaction-schema changes.** Progress is just `account.balanceMinor - goal.startedMinor` (the account's own cached balance, already correct and already maintained by every existing money-writing code path).
- **`tagged` mode needs zero transaction-schema changes either.** `transactions.tags` (text array) already exists (`apps/api/src/common/db/schema/transaction.ts:35`) and is already settable on both regular transactions and transfers (`CreateTransactionSchema.tags`, `CreateTransferSchema.tags` ‚Äî `packages/shared/src/transaction.ts:21,74`). No new `goalId` FK column is needed (unlike, say, `importBatchId`/`transferGroupId`, which exist because *many* transactions share one batch/group by construction ‚Äî a goal tag is a much looser, user-chosen label, and reusing `tags` avoids a migration entirely).

So the **entire backend addition is one new table (`goals`) + module** ‚Äî no changes to `accounts` or `transactions` schemas at all.

## Data model (new Drizzle migration)

New table `goals` (`apps/api/src/common/db/schema/goal.ts`, new file, registered in `schema/index.ts`):

```
id uuid PK
userId text FK -> user
name text
targetMinor bigint                -- must be > 0
targetDate timestamptz nullable   -- omitted = "at current rate" mode
fundingMode goal_funding_mode enum ('linked_account' | 'tagged')
linkedAccountId uuid nullable FK -> accounts
tag text nullable
priority integer                  -- server-assigned, append-to-end on create; reassigned via reorder endpoint
status goal_status enum ('active' | 'achieved' | 'abandoned')
startedMinor bigint                -- progress baseline at creation, so a linked account's pre-existing balance doesn't inflate progress
createdAt, updatedAt timestamptz
```

Indexes/constraints:
- `uniqueIndex("goals_user_id_tag_unique").on(userId, tag).where(tag IS NOT NULL)` ‚Äî one goal per tag per user (avoids two goals silently sharing a contribution pool).
- `uniqueIndex("goals_linked_account_id_unique").on(linkedAccountId).where(status = 'active' AND linkedAccountId IS NOT NULL)` ‚Äî an account can only back one *active* goal at a time (an abandoned/achieved goal frees its account up for reuse).

New enums in `enums.ts`: `goalFundingModeEnum`, `goalStatusEnum`. Extend the existing `notificationTypeEnum` with a new value `'goal_achieved'` (additive `ALTER TYPE ... ADD VALUE`, generated by `pnpm migrate:generate` like every other schema change ‚Äî AGENTS.md ¬ß4/¬ß7).

**Immutability rule**: `fundingMode` and `linkedAccountId`/`tag` are set once at creation and never changed by `PATCH` ‚Äî switching funding mode mid-goal would silently reset what "progress" means. Abandon and recreate instead if the funding approach needs to change.

## Shared schemas (`packages/shared/src/goal.ts`, new file)

- `GoalIdSchema`, `GoalFundingModeSchema = z.enum(["linked_account", "tagged"])`, `GoalStatusSchema = z.enum(["active", "achieved", "abandoned"])`.
- `CreateGoalSchema` ‚Äî `{ name, targetMinor (positive), targetDate?, fundingMode, linkedAccountId?, tag? }`, with `.refine()` cross-field rules mirroring `ColumnMappingSchema`'s pattern (`packages/shared/src/import.ts:27-49`): `fundingMode === 'linked_account'` requires `linkedAccountId` and forbids `tag`; `fundingMode === 'tagged'` requires `tag` and forbids `linkedAccountId`. `priority` is never client-supplied (server appends to end).
- `GoalSchema` ‚Äî the stored row **plus** computed `progressMinor` (not persisted ‚Äî assembled by the service on every read, same "assemble from multiple sources" style as `NetWorthService.get`).
- `UpdateGoalSchema` ‚Äî `name`/`targetMinor`/`targetDate`/`tag` patch (hand-written, not `.partial()`, for the same reason `UpdateTransactionSchema` is hand-written ‚Äî see `packages/shared/src/recurring.ts:85-91`'s comment on `.partial()` silently reapplying defaults).
- `ReorderGoalsSchema` ‚Äî `{ goalIds: GoalId[] }` (full desired order; service reassigns `priority = index`).
- `GoalPlanSchema` ‚Äî `{ goalId, mode: 'target_date' | 'at_current_rate', requiredMonthlyMinor: number | null, projectedCompletionDate: Date | null }` (see below).

## Backend module: `apps/api/src/goals/`

Mirrors the assets module's scaffolding exactly (`apps/api/src/assets/`: controller ‚Üí mutation-service ‚Üí service ‚Üí repository, idempotency keys, `withTxn`, audit records, "create entity + baseline in one transaction" pattern):

```ts
@Module({
  imports: [AccountsModule, TransactionsModule],
  controllers: [GoalController],
  providers: [GoalRepository, GoalService, GoalMutationService, GoalsProgressCron]
})
export class GoalsModule {}
```

- **`goal.repository.ts`** ‚Äî CRUD, priority reassignment, and the two progress-source queries:
  - `sumTaggedContributions(userId, tag)`: `SELECT sum(case type='income' then amount else -amount end) WHERE userId=? AND status='posted' AND tag = ANY(tags)` ‚Äî symmetric signed-sum, same shape as every other rollup/balance query in the codebase. No new index needed at personal-finance data volumes (a GIN index on `tags` is a documented follow-up if it's ever needed).
  - Linked-account progress needs no new query at all ‚Äî `AccountRepository.list`/an existing single-account lookup already returns `balanceMinor`.
- **`goal.service.ts`**:
  - `createInTx(userId, input, tx)` ‚Äî inserts the goal row with server-computed `startedMinor` (linked_account mode: current `account.balanceMinor` at creation time; tagged mode: `0`) and `priority` (max existing + 1), audit record ‚Äî same shape as `AssetService.createInTx`.
  - `getProgress(userId, goal)` ‚Äî `linked_account`: `account.balanceMinor - goal.startedMinor`; `tagged`: `sumTaggedContributions(userId, goal.tag)`.
  - `list`/`get` ‚Äî assemble `GoalSchema` (row + computed `progressMinor`) per goal.
  - `abandonInTx` ‚Äî mirrors `AssetService.closeInTx`.
  - `reorderInTx` ‚Äî bulk priority reassignment from `ReorderGoalsSchema.goalIds`.
  - `getPlan(userId, goal)` ‚Äî the scoped-down replacement for the original feasibility calculator:
    - If `targetDate` is set: `requiredMonthlyMinor = (targetMinor - progressMinor) / monthsRemaining(targetDate)`, `mode: 'target_date'`.
    - If no `targetDate`: compute an average monthly contribution rate since the goal's `createdAt` (`progressMinor / monthsSinceCreated`, floored at 1 month) and project `projectedCompletionDate = now + ceil((targetMinor - progressMinor) / averageMonthlyRate)` months; `null` if the rate is ‚â§ 0 ("at this rate, never" ‚Äî the UI's job to phrase, not the API's). `mode: 'at_current_rate'`.
    - This is deliberately simple (since-inception average, not a smarter trailing-window rate) ‚Äî flagged as a natural follow-up once there's appetite for it, not a blocker for shipping.
- **`goal-mutation.service.ts`** ‚Äî idempotency wrapper for `create`, `abandon`, `reorder` (mirrors `AssetMutationService` exactly, using `IdempotencyPostgresService.execute`).
- **`goals-progress.cron.ts`** ‚Äî nightly `@Cron("0 2 * * *", { timeZone: "Asia/Kolkata" })` (or a couple of minutes offset from the existing 02:00 jobs), same `if (this.config.env.SERVICE_ROLE !== "worker") return;` guard as `RollupsRefreshService`/`RecurringMaterializeService`. For every `active` goal: compute progress; if `progressMinor >= targetMinor`, flip `status` to `'achieved'` and `NotificationOutboxRepository.enqueue(userId, 'goal_achieved', { goalId, name, targetMinor }, tx)` inside the same `withTxn` as the status flip (AGENTS.md ¬ß4: outbox writes always share the triggering transaction).

## Endpoints (`goal.controller.ts`, `/v1/goals`, existing conventions ‚Äî inline `Schema.parse`, no `ZodValidationPipe`)

| Method | Path | Purpose |
|---|---|---|
| POST | `/v1/goals` | create (idempotent) |
| GET | `/v1/goals?status=active` | list, with computed `progressMinor` per goal |
| GET | `/v1/goals/:goalId` | detail |
| PATCH | `/v1/goals/:goalId` | update name/targetMinor/targetDate/tag |
| PATCH | `/v1/goals/reorder` | bulk priority reassignment |
| POST | `/v1/goals/:goalId/abandon` | idempotent, mirrors `asset.close` |
| GET | `/v1/goals/:goalId/plan` | `GoalPlanSchema` ‚Äî required-monthly or projected-completion-date |

## Testing

- Unit: cross-field `CreateGoalSchema` validation (both funding modes, both invalid combinations); progress computation for each funding mode; plan computation (`target_date` vs `at_current_rate` modes, including the "rate ‚â§ 0 ‚Üí null" case); priority reassignment.
- Integration (testcontainers): creating a `linked_account` goal against an account with an existing balance doesn't inflate initial progress (baseline capture); tagged-mode progress sums correctly across multiple tagged transactions/transfers and ignores untagged ones; the unique-tag and one-active-goal-per-account constraints are enforced; the nightly cron flips status and enqueues exactly one `goal_achieved` notification when progress crosses target, and is idempotent under concurrent/retried runs (parallel-execution test, per AGENTS.md ¬ß7).
- Bruno collection: `apps/api/bruno/goals/` mirroring existing conventions.
- Definition of done: `pnpm lint && pnpm typecheck && pnpm test && pnpm test:integration && pnpm verify:migrations && pnpm gen:client` before `pnpm build`.

## Suggested implementation order

1. `goals` table + shared schemas + module skeleton + plain CRUD (create/list/get/update/abandon/reorder) ‚Äî no progress computation yet.
2. Progress computation for both funding modes (`getProgress`) wired into list/get responses.
3. `GoalPlanSchema`/`getPlan` endpoint.
4. New `goal_achieved` notification type + nightly cron.
5. (Companion PR) Frontend `features/goals/` ‚Äî see the sibling "goal tracking ‚Äî frontend" design doc.
